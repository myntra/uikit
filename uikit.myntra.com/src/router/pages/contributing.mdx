# Contributing to UIKit

## :notebook: Our contribution model

UIKit is a UI components library using at Myntra. We pride ourselves in open and inclusive design and development. 
If you're wondering more about our contribution process, you're in the right place. First off, thank you for your interest! 
This project is made possible not just by the core UIKit team members, but also by several folks who have invested their 
own time to give back to the UIKit community.

## :white_check_mark: Prerequisites

Before contributing to UIKit, you should make sure you have the following tools installed:
- [Node.js](https://nodejs.org/en/download/) v10 or above or follow their installation through a package manager
  [here](https://nodejs.org/en/download/package-manager/))
- If you're on macOS, we recommend using [`nvm`](https://github.com/nvm-sh/nvm) to help manage different versions of
    Node.js.
- Git
- [pnpm](https://pnpm.js.org/en/installation)

You'll also need a code editor to make changes to UIKit. There are many to choose from but some popular options are
[VSCode](https://code.visualstudio.com/), [Atom](https://atom.io), and
[Sublime](https://www.sublimetext.com/).

## :pencil: Start contributing

### 1. Clone the repo:

Goto [UIKit's repository on Bitbucket](https://bitbucket.org/myntra/uikit/src/develop/) and clone the repository.

In your terminal:

```sh
git clone git@bitbucket.org:myntra/uikit.git
cd uikit
```

### 2. Work in a branch

When contributing to UIKit, your work should always be done in a branch off of your repo, this is also how you will submit 
your pull request when your work is done.

To create a new branch, ensure you are in your forked branch in your terminal and run:

```sh
git pull origin master
git checkout -b {your-branch-name}
```

### 3. Build and start the development server

From the root directory, run:

```sh
npm run clean
pnpm install --frozen-lockfile
npm run build
```

You can start with the development now :heart: :star: :boom:

### 4. Build and start the UIKit web

```sh
npm run docs:dev
``` 
This will start a development server where you can see any changes you are making to components.

To run only single component 
```sh
npm start {component-name}
``` 

### 5. Add a new components

To add a new component, create an empty folder under the components folder and run:

```sh
npm run refresh
```

This command will run the bootstrap scripts and create all necessary files required.

**Process to add a new component**
- Create a request to design team with all the necessary details.
- Get the approval from them and sign off from respective product managers.
- Create a new task on UI pod jira board and you can start your development. 

### 5. Test your JavaScript code

If you're contributing to our JavaScript code, test your changes by running our
test commands:

```sh
npm run test
```

### 7. Make a pull request

To do so, go to your terminal and run:

```sh
git add -A
git commit -m "YOUR  COMMIT MESSAGE HERE"
```

#### Commit tip

> **Writing commit messages**
>
> - `<type>` indicates the type of commit that's being made. This can be:
>   `feat`, `fix`, `perf`, `docs`, `chore`, `style`, `refactor`
> - `<scope>` The scope could be anything specifying place of the commit change
>   or the thing(s) that changed.
>
> **Commit message format:**

After your changes are commited, run:

```sh
git push -u origin { YOUR_BRANCH_NAME }
```

Once your code is pushed, create a pull request and send it to the UIKit team to review it.

## :raising_hand: FAQ

### Who can contribute?

Anyone! We mean it.

- **Development:** If coding is your thing, you can help us by contributing bug
  fixes or community components.
- **Design:** Design contributions can vary from visual assets, UX interactions,
  motion design, Sketch kit bug fixes and more. Changes to the core design of the component are not adviseble unless 
  approved from the design team
- **Content:** Our documentation is just as important as our design and code
  assets. Whether it's updating our current docs, or adding new use cases as an example.
- **Research:** If you're a researcher and have
  findings that you think could improve UIKit's users' experience, you're in the
  right place. This kind of contribution is most effective if coupled with
  design and development forces.

### What is the contribution process?
With that all in place, you're ready to start contributing to Carbon!

1. **Issue:** Check for UI Pod's jira dashboard for an _existing_ issue related to your contribution. If none exists, 
   open a new issue under UI Oncall epic of the current quater. We reserve the right to close any issues that haven't been 
   filled out properly according to the issue template.
4. **Pull request:** Submit a PR. Be sure to fill out all the details properly.
5. **Approval:** Get PR approved by design and developers, or make any necessary
   changes for approval. This process may be quick or take a few iterations of
   feedback-update.
6. **Documentation:** After design and dev have approved and merged PR, update
   any website documentation if necessary. One of the best examples for this is
   if you're contributing to component work which has website documentation
   related to your contribution.

## :new: What's next on UIKit?

Some ideas for contributing to UIKit.

### Extract sub-component doc comments

Extract doc comment for sub-component (e.g. `TableColumn`) and put in generated browser types
file (prepend to `Table.Column`).


### Preserve all type definitions

Keep all local type definitions from components.

### CDN for READY state components

Create a webpack plugin which transforms imports to CDN URLs.

e.g. if an application is using `Button` UIKit v1.5.2, we can load it from
`https://cdn.uikit.myntra.com/component/button@1.5` or `https://cdn.uikit.myntra.com/component/button@1`.

Benefits of this approach:

- Long term cross application caching.
- Automated patch (bug fixes) shipping.

  Depending on minor verion (`button@1.5`) would get latest patch (`button@1.5.3`) - _so bug fixes are received in an application without re-deployment_.

  Depending on major verion (`button@1`) would get latest minor (`button@1.6.0`) - _so new features are received in an application without re-deployment_.

### Ability to submit contents of embedded editor as an example for active component

A trigger action that takes contents of the embedded editor, asks for some description and generates a PR to docs.

## What's next on Spectrum?

Some ideas for contributing to Spectrum.

### `<AutoBreadCrumbs>` component
  Find all registered paths matching current route and generate breadcrumbs.

  e.g.: Routing scenario
  ``` yaml
  currentPath: `/components/button`
  registeredPaths: [
    `/` # title: Home
    `/components` # title: Component
    `/components/_name` # title: (name) => components[name]
  ]
  breadcumbs: [
    Home # from `/`
    Component # from '/components'
    Button # from current page
  ]
  ```

### `AutoNavBar>` component
  Find all route pages. Use directory hierarchy to genearte nav.

  e.g.: Directory structure
  ``` yaml
  pages:
    - index.jsx
    - components/
      - index.jsx
      - _name.jsx
      - _name.route.js
    - spectrum/
      - index.jsx
      - _plugin/
        - _version.jsx
        - _version.route.js
  ```

  Possible Nav (with hypothetical `spectrum` API)

  ``` jsx
  <NavBar title={spectrum.appName}>
    <NavBar.Item to="/">{spectrum.route.title('/')}</NavBar.Item>
    <NavBar.Group title={spectrum.route.groupName('/components')}>
      <NavBar.Item to="/components">{spectrum.route.title('/components')}</NavBar.Item>
      {
        spectrum.route.all('/components/_name').map(
          route => (
            <NavBar.Item to={route.url}>{route.title}</NavBar.Item>
          )
        )
      }
    </NavBar.Group>
    <NavBar.Section title={spectrum.route.groupName('/spectrum')}>
      <NavBar.Item to="/spectrum">{spectrum.route.title('/spectrum')}</NavBar.Item>
      {
        spectrum.route.all('/components/_plugin').map(
          route => (
            <NavBar.Group title={route.groupName}>
              {
                route.all('/_version').map(
                  route => (
                    <NavBar.Item to={route.url}>{route.title}</NavBar.Item>
                  )
                )
              }
            </NavBar.Group>
          )
        )
      }
    </NavBar.Section>
  </NavBar>
  ```
